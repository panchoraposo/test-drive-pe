= Visión general de la arquitectura del workshop

Esta página describe la arquitectura lógica del escenario Neuralbank: cómo Developer Hub, GitOps, CI/CD, identidad, seguridad de APIs, IA y observabilidad encajan en OpenShift.

== Vista de componentes

image::02-component-view.svg[Workshop component diagram, width=100%, link=_images/02-component-view.svg]

Cada bloque tiene un rol claro: el Hub orquesta la experiencia; Git es la fuente de verdad; ArgoCD aplica el modelo declarativo; Tekton automatiza la cadena de suministro de software; el gateway y la capa de políticas exponen y protegen el tráfico; Lightspeed agrega asistencia de IA; y el stack de observabilidad monitorea todo.

== Flujo principal: del template al despliegue

image::02-scaffolding-flow.svg[Scaffolding flow from Developer Hub to deployment, width=100%, link=_images/02-scaffolding-flow.svg]

. El/la desarrollador/a abre **Developer Hub** y ejecuta un **Software Template**.
. El template **publica** un repositorio en **Gitea** con código, manifiestos, `devfile.yaml` y definición de pipeline.
. **ArgoCD** detecta la Application y **sincroniza** los manifiestos al namespace `{user_name}-neuralbank`.
. **Tekton** ejecuta un **PipelineRun** (clone, build, image, deploy) gatillado por un **webhook de Gitea**.
. OpenShift **planifica pods**, expone **Services** y crea recursos de **Gateway API**.
. El componente queda **registrado** en el catálogo de Developer Hub.

== Patrón Connectivity Link

image::02-connectivity-link.svg[Connectivity Link pattern with Gateway API and Kuadrant, width=100%, link=_images/02-connectivity-link.svg]

El patrón **Connectivity Link** usa **Gateway API** con políticas de **Kuadrant** para asegurar servicios expuestos:

* **Gateway** — punto de entrada de tráfico (host, listeners, TLS)
* **HTTPRoute** — hostnames y reglas de ruteo hacia Services backend
* **AuthPolicy** — autenticación vía OIDC (JWT de Keycloak) o API Key (header `X-API-Key`)
* **RateLimitPolicy** — límites de tasa para proteger backends

=== Estructura YAML de AuthPolicy

[source,yaml,subs="attributes+"]
----
apiVersion: kuadrant.io/v1
kind: AuthPolicy
metadata:
  name: neuralbank-backend-auth
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: neuralbank-backend-route
  defaults:
    rules:
      authentication:
        api-key-auth:
          apiKey:
            allNamespaces: true
            selector:
              matchLabels:
                app: neuralbank-backend
                kuadrant.io/apikey: "true"
          credentials:
            customHeader:
              name: X-API-Key
          priority: 0
        oidc:
          jwt:
            issuerUrl: https://rhbk.{cluster_domain}/realms/neuralbank
          credentials:
            authorizationHeader:
              prefix: Bearer
          priority: 1
----

== MCP gateway de Red Hat Connectivity Link (Technology Preview)

El link:https://docs.redhat.com/en/documentation/red_hat_connectivity_link/1.3/html/installing_the_mcp_gateway/mcp-gateway-install[MCP gateway de Red Hat Connectivity Link] (Technology Preview) agrega servidores **Model Context Protocol** (MCP) detrás de un endpoint de Gateway API. El operador se instala desde el catálogo `redhat-operators` (canal `preview`). OpenShift Lightspeed y otros asistentes de IA lo usan para invocar herramientas del cluster de forma segura:

    OCP -->|oc| CLUSTER

== Rol de cada componente

[cols="1,3",options="header"]
|===
|Componente |Rol

|Developer Hub |Portal de desarrolladores: catálogo, templates, plugins, chat con IA
|Gitea |Git interno: código fuente, manifiestos, triggers de pipeline
|ArgoCD |Sync GitOps desde Git al estado del cluster
|Tekton |Ejecución de pipelines: clone, build, image, deploy
|Keycloak |Identidad y SSO; alimenta políticas OIDC y acceso al Hub
|Dev Spaces |IDE en la nube basado en `devfile.yaml`, conectado al mismo repo
|Gateway API / Kuadrant |Ingreso de tráfico, ruteo y políticas de auth/rate-limit
|OpenShift Lightspeed |Asistente de IA con MCP Gateway para operaciones del cluster
|LiteLLM |Proxy de LLM para OpenShift Lightspeed
|Grafana |Dashboards de monitoreo y alertas
|Service Mesh (consola OpenShift) |Grafo de tráfico y trazas distribuidas en la consola web
|Mailpit |Testing de emails para notificaciones
|===
