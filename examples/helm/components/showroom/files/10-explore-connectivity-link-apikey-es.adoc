= Autenticación por API Key (NFL Wallet)

En este módulo explorarás cómo **Red Hat Connectivity Link** protege la aplicación **NFL Wallet** pre-desplegada usando **AuthPolicy con API Key** y **RateLimitPolicy**.

== Arquitectura

NFL Wallet está desplegado en el namespace `nfl-wallet-prod`. Usa el mismo backend Quarkus pero está protegido por **API Key** en lugar de OIDC, lo que lo hace ideal para acceso **máquina a máquina (M2M)**.

[mermaid]
....
sequenceDiagram
    participant Client
    participant Gateway as Istio Gateway
    participant Auth as Kuadrant AuthPolicy
    participant RL as RateLimitPolicy
    participant Backend as NFL Wallet Backend

    Client->>Gateway: GET /api/v1/customers
    Gateway->>Auth: Check X-API-Key header
    alt No Key or Invalid Key
        Auth-->>Client: 401 Unauthorized
    else Valid Key
        Auth->>RL: Check rate limit (120 req/min)
        RL->>Backend: Forward request
        Backend-->>Client: 200 JSON response
    end
....

== OIDC vs API Key: cuándo usar cada uno

[cols="1,1,1",options="header"]
|===
| | OIDC (Neuralbank) | API Key (NFL Wallet)

| **Mecanismo de auth** | Tokens JWT de Keycloak | API Key en header HTTP
| **Header** | `Authorization: Bearer <token>` | `X-API-Key: <key>`
| **Respuesta no autorizada** | Redirect 302 al login | Error JSON 401
| **Ideal para** | Usuarios interactivos (apps web) | Integraciones M2M, scripts, CI/CD
| **Gestión de credenciales** | Keycloak emite tokens | Secrets de Kubernetes con labels
|===

== Inspeccionar recursos

[.console-input]
[source,bash]
----
oc get pods -n nfl-wallet-prod
oc get gateway -n nfl-wallet-prod
oc get httproute -n nfl-wallet-prod
oc get authpolicy -n nfl-wallet-prod -o wide
oc get ratelimitpolicy -n nfl-wallet-prod -o wide
----

Verifica que la AuthPolicy muestre `ACCEPTED: True` y `ENFORCED: True`.

== AuthPolicy con API Key

[source,yaml]
----
apiVersion: kuadrant.io/v1
kind: AuthPolicy
metadata:
  name: nfl-wallet-auth
  namespace: nfl-wallet-prod
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: nfl-wallet-route
  defaults:
    rules:
      authentication:
        api-key-auth:
          apiKey:
            allNamespaces: true
            selector:
              matchLabels:
                app: nfl-wallet
                kuadrant.io/apikey: "true"
          credentials:
            customHeader:
              name: X-API-Key
      response:
        unauthenticated:
          code: 401
          body:
            value: '{"error":"API Key required. Include X-API-Key header."}'
----

== Listar API Keys disponibles

Las API Keys se almacenan como Secrets de Kubernetes con labels que detectan Kuadrant/Authorino:

[.console-input]
[source,bash]
----
oc get secrets -n nfl-wallet-prod -l kuadrant.io/apikey=true

# Decodificar la key admin (permite POST/PUT/DELETE)
oc get secret nfl-wallet-apikey-admin -n nfl-wallet-prod \
  -o jsonpath='{.data.api_key}' | base64 -d ; echo

# Decodificar la key readonly (solo GET)
oc get secret nfl-wallet-apikey-readonly -n nfl-wallet-prod \
  -o jsonpath='{.data.api_key}' | base64 -d ; echo
----

== Prueba: request sin API Key (espera 401)

[.console-input]
[source,bash,subs="attributes+"]
----
curl -s -o /dev/null -w "HTTP Status: %\{http_code}\n" \
  https://nfl-wallet.{cluster_domain}/api/v1/customers
----

[.console-output]
----
HTTP Status: 401
----

== Prueba: con API Key válida (espera 200)

[.console-input]
[source,bash,subs="attributes+"]
----
ADMIN_KEY=$(oc get secret nfl-wallet-apikey-admin -n nfl-wallet-prod \
  -o jsonpath='{.data.api_key}' | tr -d '\n' | base64 -d)

curl -s -H "X-API-Key: $ADMIN_KEY" \
  "https://nfl-wallet.{cluster_domain}/api/v1/customers" \
  | jq '.content[:1]'
----

== Prueba: crear un cliente (POST)

[.console-input]

== Prueba rápida de rate-limit

Esta API también tiene rate limiting habilitado (vía Kuadrant PlanPolicy). La key **readonly** está en el plan **free** (**10 requests por minuto**), así que puedes gatillar `429` rápidamente:

[.console-input]
[source,bash,subs="attributes+"]
----
READONLY_KEY=$(oc get secret nfl-wallet-apikey-readonly -n nfl-wallet-prod \
  -o jsonpath='{.data.api_key}' | tr -d '\n' | base64 -d)

for i in $(seq 1 12); do
  code=$(curl -s -o /dev/null -w '%\{http_code}' \
    -H "X-API-Key: $READONLY_KEY" \
    "https://nfl-wallet.{cluster_domain}/api/v1/customers")
  echo "Request $i: HTTP $code"
done
----

Las primeras 10 deberían retornar `200`. Las requests 11+ deberían retornar `429 Too Many Requests`. Si no ves `429`, espera 60 segundos y reintenta.

== Cómo funciona la autenticación por API Key

[mermaid]
....
sequenceDiagram
    participant Client
    participant Authorino as Kuadrant Authorino
    participant K8s as Kubernetes API
    participant Backend

    Note over Authorino,K8s: Authorino watches Secrets with label kuadrant.io/apikey=true
    Client->>Authorino: Request with X-API-Key header
    Authorino->>K8s: Lookup matching Secret
    alt Key Found
        Authorino->>Backend: Forward request
        Backend-->>Client: 200 JSON
    else Key Not Found
        Authorino-->>Client: 401 Unauthorized
    end
....

== Resumen

El patrón de API Key es ideal para:

* Integraciones máquina a máquina (M2M)
* Pipelines CI/CD que consumen APIs
* Partners externos con credenciales de larga duración

Ambos patrones (OIDC y API Key) usan el mismo stack: **Istio Gateway + HTTPRoute + políticas de Kuadrant**, demostrando la flexibilidad de Red Hat Connectivity Link.
